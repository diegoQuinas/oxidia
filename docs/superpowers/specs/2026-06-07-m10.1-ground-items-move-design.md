# M10.1 ‚Äî Ground items dynamic + move-thing (design)

> Open Tibia server in Rust, protocol **10.98**, client **OTClient Redemption**.
> TFS 1.4.2 (`reference/tfs/`) is the spec oracle ‚Äî read it, never port line-by-line.
> Architecture: single-actor world (`world::game::GameWorld`), one writer, no locks.

## Chain context

M10 (Inventory & equipment) is split into a dependency-ordered chain, mirroring
the M6.1 / M7.1 precedent and the ~400-line PR budget:

| # | Slice | State |
|---|-------|-------|
| **M10.1** | **Ground items dynamic + move-thing (the mutable foundation)** | this doc |
| M10.2 | Inventory & equipment slots (equip/unequip, pickup/drop, weight/cap, persist) | planned |
| M10.3 | Containers (backpack) + use-to-open (`0x82` use that opens a container) | planned |

"Use items" is **not** a fourth slice: in TFS the real `use` content (levers,
keys, runes, potions) is `actions.xml` ‚Üí Lua. The only non-Lua use is "double-click
a container opens it", which lives in M10.3. The rest defers to **M11 (Lua runtime)**
‚Äî the same call made for M6.2 (ladders use-driven ‚Üí M11).

## Why this is the foundation

Today the entire item world is **immutable**: `StaticMap` lives in an `Arc`, with
zero mutable access at runtime (the only mutation is `load_item_metadata` at boot).
M10.1's job is to introduce a **mutable item layer** that everything downstream ‚Äî
inventory (M10.2), containers (M10.3), monster spawns (M12), loot/corpses (M13) ‚Äî
hangs off. Get this spine right and the rest is plumbing onto it.

## Scope

Move items that are **already on the map**, tile to tile:

- Ground‚Üíground move of an existing map item (the `0x78` throw protocol, both
  endpoints are real tile positions).
- **Merge** stackables (drop gold onto a gold pile ‚Üí counts sum, capped at 100).
- **Split** a stack (the client sends `count` in the `0x78`; honoring it is free ‚Äî
  the count dialog is client-side).
- **Broadcast** the tile change to spectators (`0x6A` add / `0x6B` update / `0x6C`
  remove, item forms).
- **Faithful range** ‚Äî full `Map::canThrowObjectTo` port: throw range + line-of-sight
  (Bresenham `isSightClear`), verified against `map.cpp`.

### Out of scope (later in chain / later milestones)

- Inventory / container endpoints of `0x78` (the `0xFFFF` special positions) ‚Üí M10.2 / M10.3.
- Drop-from / pick-up-to inventory ‚Üí M10.2 (needs body slots).
- Auto-walk to an out-of-reach source tile (TFS `playerAutoWalk` then retry) ‚Üí
  deferred with the rest of auto-walk (deferred since M4); M10.1 **rejects** a move
  whose source is not already in reach.
- Item decay / despawn timers ‚Üí its own concern (no item timers exist yet).
- Persistence of dynamic ground items ‚Üí **none**: TFS does not persist dropped
  ground items either; the overlay is in-memory and resets on restart (faithful).
- Weight / capacity enforcement ‚Üí M10.2 (item stays in the world, nobody carries it).

## Architecture (locked: overlay copy-on-write)

`StaticMap` stays **immutable** ‚Äî ground, `blocked`, `floor_change`,
`protection_zone`, `item_meta`, and the original static tile stacks are untouched.

The actor (`Game`) gains a **sparse copy-on-write overlay**:

```
dynamic: HashMap<(u16,u16,u8), TileStack>
```

Only tiles touched at runtime get an entry. On the first mutation of a tile,
`materialize(pos)` clones the static `TileStack` into `dynamic`, then the change is
applied to the owned copy. Untouched tiles are never cloned (minimal memory).

**The encoder does not change.** The M3 map encoder already reads tiles through the
`TileSource` trait (`tile()` + `creature_stackpos()`, built map-agnostic on
purpose). M10.1 adds a merged view:

```rust
struct MergedTiles<'a> { dynamic: &'a HashMap<(u16,u16,u8), TileStack>, base: &'a StaticMap }
// impl TileSource: check `dynamic` first, fall back to `base`.
```

`do_move` / `do_look` / the login burst pass `MergedTiles` (built from `&self.dynamic`
+ `&*self.map`) wherever they currently pass `&*self.map`. Spectators who walk a
modified tile into view get the dynamic stack in their map slice; everyone else is
unaffected.

### Stackpos invariant survives

M5 established ‚â§1 creature per tile so the static `creature_stackpos`
(= `pre_creature_len`) is always correct. Dropped/moved items are **"down" items** ‚Äî
in TFS tile order they come **after** creatures (`ground ‚Üí top items ‚Üí creatures ‚Üí
down items`). Adding or removing a down item therefore does **not** shift the
creature's stackpos. The M5 invariant holds for the common case.

`pre_creature_len` only changes if a **top item** (`alwaysOnTop`) is added at
runtime ‚Äî and movable/pickupable items are down items, not top items. M10.1 only
moves movable items, so `pre_creature_len` is stable and the incremental
`0x6A`/`0x6C` stackpos math stays correct.

## A. Data layer (`formats`)

`ItemType::is_moveable()` over the existing `flags` word ‚Äî `FLAG_MOVEABLE`
(`items.otb` flag bit; verify the bit index against `items.otb`/TFS `ITEM_ATTR`
during the spike), a method (no new field, no struct-literal churn), mirroring the
existing `is_pickupable()` / `is_stackable()`.

## B. World layer (`world`)

### Overlay + merged view (map.rs / game.rs)

- `Game.dynamic: HashMap<(u16,u16,u8), TileStack>` (empty by default).
- `Game::materialize(&mut self, pos)` ‚Äî COW clone of the static stack on first touch.
- `MergedTiles<'a>` implementing `TileSource` (overlay-first lookup). `TileStack`
  becomes `pub(crate)` + `Clone` so the actor can own materialised copies and
  `MergedTiles` (same module) can read its fields.

### `Command::MoveThing` + `do_move_thing` (game.rs)

`Command::MoveThing { id, from: Position, from_stackpos: u8, to: Position, count: u8 }`:

1. **Resolve** the thing at `(from, from_stackpos)` ‚Äî reuse the `do_look` stackpos
   resolution (ground ‚Üí top ‚Üí creature ‚Üí down). Must resolve to a tile **item**
   (moving a creature is not a `0x78` action). Out-of-range / no-item ‚Üí drop silently.
2. **Moveable check** ‚Äî `item_meta`/`is_moveable()`; reject non-moveable (push the
   TFS "You cannot move this object." cancel ‚Äî `0xB4` text, same shape as the M7 PZ
   reject, message type to be confirmed against `protocolgame.cpp`).
3. **Reach check** ‚Äî player must be in range of `from` (TFS
   `Position::areInRange<1,1,0>(playerPos, from)`); if not, reject (no auto-walk).
4. **Throw check** ‚Äî `canThrowObjectTo(from, to)` faithful port: distance ‚â§ throw
   range on each axis **and** `isSightClear` (Bresenham LOS). Verify the range
   constants and the LOS walk against `map.cpp` during the spike.
5. **Destination validity** ‚Äî `to` has ground and does not block the item; respect
   the 10-thing cap (TFS drops to the tile below / rejects when full ‚Äî port the
   minimal faithful behavior).
6. **Apply** (COW on both `from` and `to`):
   - Remove `count` from the source item. Partial (`count < stack`) ‚Üí source becomes
     a split remainder (`0x6B` update at source); full ‚Üí remove (`0x6C` at source).
   - Add to dest: if the top down-item is the same stackable type, **merge** up to
     100 (overflow spills to a new stack ‚Äî port TFS `internalAddItem` stacking);
     else append a new down item (`0x6A` add at dest).
7. **Broadcast** each source/dest tile change to its spectators (see C).

The function mirrors `do_move`'s shape: read state, validate, commit, broadcast to
the spectator union. Single packet builder intact, no new locks.

## C. Protocol layer (`protocol`)

New / extended **pure-wire** encoders (no game logic):

- `parse_throw(body) -> Option<(Position from, u16 sprite, u8 from_stackpos,
  Position to, u8 count)>` ‚Äî inbound `0x78` (`parseThrow`, `protocolgame.cpp:509`).
  Wire: `[from x u16,y u16,z u8][sprite u16][from_stackpos u8][to x u16,y u16,z u8][count u8]`.
- `add_tile_item(pos, stackpos, item: &WireItem) -> Vec<u8>` ‚Äî item form of `0x6A`
  (`[0x6A][pos][stackpos][u16 client_id][u8 count?]`). The existing
  `add_tile_creature` is the creature form of the same opcode; factor the shared
  `[0x6A][pos][stackpos]` header so both reuse it.
- `update_tile_item(pos, stackpos, item: &WireItem) -> Vec<u8>` ‚Äî `0x6B` positional
  form (`sendUpdateTileItem`, count change on merge/split). Note `0x6B` is also
  `creature_turn` (id form) ‚Äî same opcode, namespaced by content, exactly like the
  M4/M5 inbound/outbound overlaps.
- `remove_tile_thing(pos, stackpos)` ‚Äî `0x6C` positional form **already exists**
  (used for creatures in M5); reused as-is for item removal.

The single-item wire shape (`[u16 client_id][u8 0xFF]` + optional count) already
lives in `map_description.rs::add_item`; expose/share it so the `0x6A`/`0x6B` item
encoders emit identical bytes.

## D. Service wiring (`server`)

`reader_loop` (game_service.rs): one new guard before the walk/turn fallthrough
(slots in after the `0x8D` look-battle block) ‚Äî `0x78` ‚Üí `look`-style parse via
`protocol::move_thing::parse_throw` ‚Üí `world.move_thing(id, from, from_stackpos,
to, count)`. `0x78` is currently silently drained, so no behavior regression.

## E. Implementation order ‚Äî spike first, tests deferred

**This project validates live before writing tests** (workflow: "No tests until
manual validation"; M9 shipped the same spike-first way). M10.1 follows suit:

1. **Spike** the full vertical slice ‚Äî overlay + merged view, `parse_throw`, the
   `0x6A`/`0x6B`/`0x6C` item encoders, `do_move_thing` with the faithful
   `canThrowObjectTo` + LOS port, and the reader guard ‚Äî wired end to end against
   the real client.
2. **Live check (manual gate, by the roadmap owner):** real OTClient ‚Äî drag a movable
   ground item to another tile; drag a gold pile onto another ‚Üí counts merge; split a
   stack (drag with a count) ‚Üí two piles; a non-movable item refuses to move; a second
   client sees every change live with no desync; moving across the screen respects
   line-of-sight (cannot throw through a wall).
3. **Tests (only after the live check passes):** `formats` `is_moveable`; `protocol`
   `parse_throw` round-trip + the `0x6A`/`0x6B` item encoders; `world` `do_move_thing`
   ‚Äî ground‚Üíground move, merge sums counts, split leaves two stacks, non-movable
   rejected, out-of-throw-range / LOS-blocked rejected, spectator broadcast. Then the
   standing gate: `cargo test` green (workspace), `cargo clippy --all-targets -- -D
   warnings` clean, `#![forbid(unsafe_code)]` intact.

## Deferred (summary)

- Inventory / container `0x78` endpoints, pickup/drop, weight/cap ‚Üí M10.2 / M10.3.
- Auto-walk to an out-of-reach source ‚Üí with the rest of auto-walk (since M4).
- Item decay / despawn; persistence of dynamic ground items.
- `use` content (levers/keys/runes/potions) ‚Üí M11 (Lua).
