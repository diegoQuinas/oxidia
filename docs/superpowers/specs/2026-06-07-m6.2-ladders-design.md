# M6.2 — Ladders & holes (use-driven floor changes)

**Status:** DEFERRED to M11 (Lua runtime / `onUse` hooks). Not implemented as a
standalone Rust milestone. This document records the TFS research and the
decision rationale so M11 can implement it faithfully.
**Date:** 2026-06-07
**Depends on:** M11 (embedded Lua via `mlua`, `onUse` content hooks).

## Why deferred (decision)

The investigation below shows that use-driven floor changes are **entirely
script-driven behavior**, not data. Neither the ladder (`1386`) nor the sewer
grate (`430`) carry any attribute in `items.xml` — both are empty `<item>`
entries. Their behavior lives 100% in `data/actions/actions.xml` (registration)
plus `data/actions/scripts/other/teleport.lua` (the `onUse` logic): a hardcoded
ladder-id table, `moveUpstairs`, and `teleportTo`.

The only *data-driven* floor-change signal in `items.xml` is the `floorchange`
attribute on ramps/holes — and that is **walk-driven**, already consumed by
M6.1's `do_move` vertical path.

Implementing M6.2 standalone would mean hardcoding a Lua script's behavior (an
item-id table + `move_upstairs`) into the Rust core — exactly the kind of
mutable, scriptable content the locked architecture assigns to embedded Lua.
The roadmap already plans M11 as "Lua runtime (mlua): hot-reloadable content
hooks (`onUse`/`onStepIn`/`onSay`/…)". `teleport.lua` IS an `onUse` hook, so it
belongs there. Deferring avoids baking scriptable behavior into Rust and avoids
building the inbound `0x82` parser speculatively before a consumer exists.

When M11 lands, this mechanic is a near-verbatim port of `teleport.lua` running
on the Lua runtime, calling into the M6.1 vertical engine and a teleport
broadcast primitive (see "Architecture" and "Broadcast" below for the shapes
those primitives need to expose to Lua).

## Goal (for M11)

Port TFS use-driven floor changes: using a ladder climbs you one floor up;
using a hole/grate drops you one floor down. This is the *use-driven* half of
floor changes (M6.1 delivered the *walk-driven* half). The entire mechanic in
TFS lives in one action script — `data/actions/scripts/other/teleport.lua` —
registered for a small fixed set of item ids in `data/actions/actions.xml`.

## Source of truth (TFS)

`teleport.lua` is the complete mechanic:

```lua
local upFloorIds = {1386, 3678, 5543, 22845, 22846}
function onUse(player, item, fromPosition, target, toPosition, isHotkey)
    if table.contains(upFloorIds, item.itemid) then
        fromPosition:moveUpstairs()       -- ladder -> up
    else
        fromPosition.z = fromPosition.z + 1   -- grate/hole -> down
    end
    if player:isPzLocked() and Tile(fromPosition):hasFlag(TILESTATE_PROTECTIONZONE) then
        player:sendCancelMessage(RETURNVALUE_PLAYERISPZLOCKED)
        return true
    end
    player:teleportTo(fromPosition, false)
    return true
end
```

`actions.xml` registers `teleport.lua` for itemids: `430, 1369, 1386, 3678,
5543` (the script's `upFloorIds` also lists `22845, 22846`, ladder ids in a
`fromid` range).

**There is no item transform** (no closed-grate -> open-hole). Using the item
teleports directly. Walk-driven holes (`floorchange="down"` items: 383, 392,
469, …) are a separate mechanic already handled by M6.1.

`Position:moveUpstairs()` (`data/lib/core/position.lua`):

```
moveUpstairs():
    z = z - 1                                  # ascend one floor
    candidate = tile directly SOUTH (x, y+1, z-1)   # preferred landing
    if candidate not walkable:
        scan neighbors N, E, W, SW, SE, NW, NE (SOUTH already tried)
        land on the first walkable one
    fallback: land SOUTH even if not walkable
```

`teleportTo(pos, false)` — `false` = no push/magic-of-move; the client renders
it as a teleport (creature removed here, appears there), with a
`CONST_ME_TELEPORT` puff, not a walk step.

## Architecture

The action set uses **server item ids** (1386, 430, …). Tiles in `StaticMap`
store `WireItem { client_id }` only — the server id is not retained per tile.
M6.1 solved the identical problem for `floor_change`: `from_formats` resolves
each item's `server_id -> ItemType` at load and caches a per-tile attribute.

**M6.2 follows that exact pattern.** At load, precompute a per-tile
`use_floor_change: Option<UseDir>` (`Up` / `Down`) derived from whether the tile
holds an item whose server id is in the teleport action set. This mirrors
`floor_change_at`, trusts the server-side map (never the client's `spriteId`),
and keeps the lookup O(1) at use time.

### Flow

```
client uses item -> inbound 0x82
  -> parse_use_item(pos, sprite_id, stackpos, index)
  -> WorldHandle::use_item -> Command::UseItem -> do_use(id, pos)
       -> map.use_floor_change_at(pos)?          # None -> not a ladder/grate, ignore
       -> Up   : move_upstairs(pos)              # z-1; SOUTH landing, else scan 8 neighbors
          Down : (x, y, z+1)                     # same x/y, descend one floor
       -> guard: has_ground(landing) else abort (no teleport, no broadcast)
       -> commit new position
       -> broadcast TELEPORT (remove + add + CONST_ME_TELEPORT puff)
```

### Landing validation

Both directions validate `has_ground(landing)` before committing. The
walk-driven path (`do_move`) reaches vertical landings with `FLAG_NOLIMIT`
(block-solid items ignored, only a real tile required); the use-driven path
keeps the same rule: `has_ground` is required, block-solid items are not
checked. If the landing has no ground, the use is aborted — no position change,
no broadcast. (TFS trusts the mapper and would teleport into the void; we add a
defensive `has_ground` guard, consistent with how `do_move` filters an
unreachable step.)

`move_upstairs` already only lands on a *walkable* neighbor (or the SOUTH
fallback), so its landing is normally valid; the guard covers the fallback case.

### Broadcast: teleport, not walk

TFS `teleportTo` is teleport semantics, not a walk animation. A relocate emits:

- **Spectators** seeing the destination: remove the creature from the old tile,
  add it at the new tile, plus a `CONST_ME_TELEPORT` magic-effect puff.
  Spectators that only saw the origin get a plain remove.
- **Mover:** a map redescription for the new position (the destination is on a
  different floor, so the client needs the new floor's slices).

This reuses existing infrastructure: the surface<->underground boundary branch
in `do_move` (`game.rs:430-444`, clean remove+add across render stacks), the
`enter_world::magic_effect(..., EFFECT_TELEPORT)` puff used on login
(`game.rs:249`), and `walk::walk_update` for the mover's own floor-aware view.

### Action table (literal port)

- **Up** (`move_upstairs`): `1386, 3678, 5543, 22845, 22846`
- **Down** (`z + 1`): `430, 1369`

## Inbound 0x82 wire format (parseUseItem)

```
pos:       x u16 (LE), y u16 (LE), z u8
sprite_id: u16 (LE)
stackpos:  u8
index:     u8
```

The server resolves the acted-upon item from `pos` via the precomputed
per-tile `use_floor_change` (not from `sprite_id`). `stackpos`/`index` are
parsed for wire fidelity but not used by the floor-change path (these items
define their tile; no container/stack disambiguation needed for M6.2).

Note: `0x82` inbound (use item) does not collide with `0x82` outbound
(`OP_WORLD_LIGHT`) — inbound and outbound opcode tables are independent.

## Testing

- `parse_use_item`: exact-bytes round-trip; truncated body rejected.
- `move_upstairs`: SOUTH landing when walkable; neighbor-scan fallback when
  SOUTH is blocked; SOUTH fallback when no neighbor is walkable.
- `use_floor_change_at`: Up tile (ladder id), Down tile (grate id), None for a
  plain tile — precompute resolves server ids correctly.
- `do_use` round-trip: using an Up tile raises the mover's z and relocates per
  `move_upstairs`; a spectator at the destination receives remove + add + the
  `EFFECT_TELEPORT` puff. Down tile descends `z+1`. A landing without ground
  aborts (no position change, no broadcast).
