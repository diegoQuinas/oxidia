# M10.2 — Inventory & Equipment Slots (design)

**Date:** 2026-06-07
**Milestone:** M10.2 (second slice of the M10 chain: M10.1 ground-items+move → **M10.2 equip slots** → M10.3 containers/backpack)
**Workflow:** spike-first — implement the vertical slice, roadmap owner validates live in OTClient, *then* write tests.

## Goal

Let a player drag an item from the ground onto a body slot (equip), back to the
ground (unequip), or between slots, with TFS-faithful slot-type validation, and
persist the equipped items across logout/login. The equipped items render in the
client paperdoll (inventory window). **Combat is out of scope** — fist attack
stays; the weapon→damage hook is left ready for a later slice.

Builds directly on the M10.1 mutable-tile overlay and the `do_move_thing`
pipeline. The move-thing wire path already forwards inventory endpoints
(`x == 0xFFFF`) and currently early-returns them — M10.2 fills that branch in.

## Scope

In:
- 10 equipment slots (TFS `CONST_SLOT_HEAD=1` … `CONST_SLOT_AMMO=10`).
- Equip (ground→slot), unequip (slot→ground), slot→slot move.
- Strict slot-type validation derived from `items.xml` `slotType` / `weaponType`.
- `0x78` SetInventory encoder; reuse `0x79` ClearInventory.
- Persistence of equipped items (load on login, save on logout).

Out (deferred, documented below):
- Container (backpack) contents and opening → **M10.3**.
- Capacity / weight tracking.
- Two-handed wield conflict (shield ⊕ two-handed weapon mutual exclusion).
- Weapon/armor affecting combat damage.

## Architecture

### 1. Item metadata: `equip_slot`

`ItemMeta` (`crates/world/src/map.rs`) gains `equip_slot: Option<EquipSlot>`,
derived once at boot from the `items.xml` attributes. The `items.xml` loader
(`crates/formats/src/items_xml.rs`) already iterates **every** `<attribute>`
child, so reading `slotType` / `weaponType` is cheap — no new pass.

`EquipSlot` is a small enum encoding which inventory slot(s) an item admits.
TFS-faithful mapping (verified against `reference/tfs/data/items/items.xml`
`slotType`/`weaponType` values and `reference/tfs/src/creature.h` slot enum):

| Source attribute (`items.xml`)            | Allowed slot (TFS `CONST_SLOT_*`) |
|-------------------------------------------|-----------------------------------|
| `slotType=head`                           | 1 head                            |
| `slotType=necklace`                       | 2 necklace                        |
| `slotType=backpack`                       | 3 backpack                        |
| `slotType=body`                           | 4 armor                           |
| `weaponType` ∈ {sword,axe,club,distance,wand} or `slotType=two-handed` | 5 right / 6 left (hands) |
| `weaponType=shield`                       | 5 right / 6 left (hands)          |
| `slotType=legs`                           | 7 legs                            |
| `slotType=feet`                           | 8 feet                            |
| `slotType=ring`                           | 9 ring                            |
| `slotType=ammo` or `weaponType=ammunition`| 10 ammo                           |

Items with no equip attribute → `equip_slot = None` → cannot be equipped
(rejected on any slot).

### 2. Player inventory state

`PlayerState` (`crates/world/src/game.rs`) gains:

```rust
inventory: [Option<InvItem>; 10],   // index 0..=9  ⇄  slot 1..=10

struct InvItem { server_id: u16, client_id: u16, count: u8, animated: bool }
```

`client_id` + `count` + `animated` are cached so the `0x78` encoder needs no
catalog lookup at send time (same shape as the tile-stack `WireItem`).

### 3. Protocol — `0x78` SetInventory

New encoder `set_inventory_slot(slot: u8, item: &InvItem) -> Vec<u8>`
(`crates/protocol/src/enter_world.rs`, alongside `empty_inventory`):

```
[0x78][slot u8][client_id u16][count/subtype u8 if stackable]
```

(mirrors a tile item: stackables carry the count byte, non-stackables don't —
same convention the tile encoder already uses).

Clearing a slot reuses `OP_INVENTORY_EMPTY (0x79)`: `[0x79][slot u8]`.

**Equip is private to the equipping player.** In protocol 10.98 held
weapons/shields and worn armor are NOT part of the on-map outfit packet — they
only appear in the paperdoll. So `0x78`/`0x79` go to the player's own push
channel only; spectators see nothing on the character sprite. The only
spectator-visible effect is on the **ground** side (item leaving/arriving a
tile), which already broadcasts via the M10.1 `do_move_thing` machinery.

### 4. Move-thing inventory routing (`do_move_thing`)

The inbound `parseThrow` body already carries `from`/`to` positions where
`x == 0xFFFF` denotes an inventory/container endpoint. Decode (verified vs
`reference/tfs/src/game.cpp:199-213` `internalGetCylinder`):

- `x == 0xFFFF` **and** `(y & 0x40) == 0` → **inventory**, slot = `y` (1..=10).
- `x == 0xFFFF` **and** `(y & 0x40) != 0` → **container** (cid = `y & 0x0F`) →
  out of scope, early-return (M10.3).

Four cases by endpoint kind:

1. **ground → slot (equip):** validate the dragged item's `equip_slot` admits
   the target slot; validate reach to the ground tile (the existing
   adjacency/`can_throw` checks); remove from ground (existing COW + `0x6C`
   broadcast); store into `inventory[slot-1]`; push `0x78` to self.
2. **slot → ground (unequip):** take from `inventory[slot-1]`; push `0x79` to
   self; add to the ground tile front down-item (existing COW + `0x6A`
   broadcast).
3. **slot → slot:** validate target slot admits the item; move within
   `inventory`; push `0x78` (dest) and `0x79` (source) to self. If the dest slot
   is occupied, swap (push `0x78` for both).
4. **ground → ground:** unchanged M10.1 path.

Validation failure (wrong slot type, or `equip_slot == None`) → reuse
`push_cannot_move(id, "You cannot equip this object.")`.

Stackables: equipping/unequipping moves the whole stack (ammo). Splitting a
stack into a slot follows the same clamp logic as the M10.1 ground path.

### 5. Persistence

New migration `crates/persistence/migrations/0004_player_inventory.sql`:

```sql
CREATE TABLE player_inventory (
    player_name TEXT    NOT NULL,
    slot        INTEGER NOT NULL,   -- 1..=10
    server_id   INTEGER NOT NULL,
    count       INTEGER NOT NULL DEFAULT 1,
    PRIMARY KEY (player_name, slot)
);
```

Flat by design: M10.3 adds a nullable `parent` column to nest container
contents without reshaping the table.

The world crate stays decoupled from `persistence` (DTO boundary, as today):
- `InitialState` gains `inventory: Vec<(u8 slot, u16 server_id, u8 count)>`.
  On login the world resolves each `server_id` through the item catalog to build
  `InvItem`s, and emits a `0x78` per occupied slot inside the enter-world burst
  (after `empty_inventory`).
- `SaveRecord` gains the same `inventory` vector; the server worker upserts the
  `player_inventory` rows on logout (delete-all-then-insert for that player_name,
  inside the existing save transaction).

### 6. Combat hook (deferred, not implemented)

`crates/world/src/combat.rs` already exposes `melee_damage`/`max_weapon_damage`
parameterized on `attack_value`/`attack_skill`. M10.2 leaves fist combat wired;
a later slice reads the equipped hand-slot weapon's attack value and calls
`max_weapon_damage` directly. No combat code changes in this milestone.

## Testing

Spike-first. Implement the slice; roadmap owner validates live in OTClient
(drag helmet from ground onto head slot → appears in paperdoll; drag back →
returns to ground; relog → still equipped; wrong slot → rejection text). Only
after live acceptance: regression tests for the `equip_slot` mapping, the `0x78`
encoder layout, the four move-thing routing cases, and the persistence
round-trip.

- `crates/server/src/*` — map `PlayerSave.inventory` ⇄ `InitialState`/`SaveRecord`.
