# M6.1 ‚Äî Floor Changes & Stairs ‚Äî Design

> Oxidia, the from-scratch idiomatic Rust Open Tibia server. Protocol **10.98**,
> client **OTClient Redemption**. TFS 1.4.2 (`reference/tfs/`) is a **spec
> reference only** ‚Äî verified, never ported line by line.

M6.1 completes core movement: the world stops being a single flat plane. Players
walk **up and down stairs** and into the **underground** (z ‚â• 8), and everyone in
range sees it. This is the walk-driven half of vertical movement; the use-driven
half (ladders, holes) is M6.2.

## Scope (locked)

**Walk-driven vertical movement, end to end:**

- **Stairs up / down** ‚Äî both Tibia mechanics: height-based slopes and
  `floorChange` staircase tiles (see "The two mechanics").
- **Underground viewport** ‚Äî when the player is at z ‚â• 8, the client receives a
  `¬±2`-floor band instead of the overground `7‚Üí0` stack.
- **Floor-change protocol** ‚Äî `0xBE` (move up) / `0xBF` (move down) for the
  moving player's own camera, plus the overground‚Üîunderground boundary handling.
- **Multi-floor presence** ‚Äî spectators on floors within the `¬±2` band see other
  players change floor (revives the band M5 deferred).
- **items.xml loader** ‚Äî a complete, generic, typed loader (decided with the
  user) that merges item attributes into `ItemType`; M6.1 consumes `floorChange`,
  later milestones consume the rest.

Out of scope (deferred): **ladders / holes** (M6.2, use-driven, opcode `0x82`);
**auto-walk / click-to-move**; **diagonal floor change** (TFS blocks it via
`!diagonalMovement` ‚Äî we match that); house/PZ tile semantics beyond floor
changes; real player persistence (M8).

## The two mechanics (verified against TFS)

Tibia changes floors two structurally different ways. M6.1 ports both.

### Mechanic A ‚Äî height (slopes), `game.cpp:792-820`

In `Game::internalMoveCreature`, for a non-diagonal player step:

- **Up:** if the *current* tile `hasHeight(3)` and the tile directly above is
  open, and the tile above the horizontal destination has ground (and isn't an
  immovable blocker or a floor-change tile), drop `destPos.z--`.
- **Down:** if `currentPos.z == destPos.z`, the destination on the same floor is
  void/groundless, and the tile one floor below `hasHeight(3)`, raise
  `destPos.z++`.

`hasHeight` is a per-item flag from the **`.otb` (`FLAG_HAS_HEIGHT`, bit 3)**; a
tile "has height N" when the summed item heights reach N (each height-flagged item
contributes 1).

### Mechanic B ‚Äî `floorChange` staircase tiles, `tile.cpp:queryDestination:732`

Stepping onto a tile flagged `TILESTATE_FLOORCHANGE*` redirects the creature to
another floor:

- `FLOORCHANGE_DOWN` ‚Üí descend to `z+1`, applying directional offsets
  (north/south/east/west) and the `_ALT` two-tile variants
  (`SOUTH_ALT`, `EAST_ALT`), plus the adjacent-tile alt lookups.
- `FLOORCHANGE` without `DOWN` (plain directional) ‚Üí ascend to `z-1` with the
  inverse offsets.

These tile flags are set from the items on the tile, and the per-item
`floorChange` direction lives **only in `items.xml`** (the `.otb`
`FLAG_FLOORCHANGE*` bits are marked unused in TFS and are not a reliable source).

## Architecture decision ‚Äî items.xml loader (locked with user)

We build the **complete** items.xml loader now, not a `floorChange`-only minimal
one. Rationale: `items.xml` is foundational infrastructure (M9 ground items, M10
inventory/equip, M7 combat all read it); the user chose to pay for it once. To
avoid scope-creep into hand-picking attributes, the parser reads **every
`<attribute>` generically into a typed model** rather than a curated subset.
M6.1's only *consumer* is `floorChange`; the rest is loaded and available.

| Concern | Choice | Why |
|---|---|---|
| XML parser | `roxmltree` (read-only DOM) | Pure Rust, zero unsafe, trivial for a one-shot boot-time parse of the ~26k-entry file; `quick-xml` (streaming) is overkill at boot |
| Keying | by `server_id` (`id`, plus `fromid`/`toid` ranges) | Matches the `.otb` `server_id ‚Üí client_id` index, so the merge is a direct join |
| Merge | `merge_items_xml(&mut items, &xml)` in `formats` | Single `ItemType` lookup downstream; no parallel attribute store |

## Components

### 1. `crates/formats/src/items_xml.rs` (new) ‚Äî items.xml loader

```rust
pub struct ItemsXml { by_server_id: HashMap<u16, ItemXmlAttrs> }

pub struct ItemXmlAttrs {
    pub floor_change: FloorChange, // M6.1 consumer
    // generic typed bag for everything else items.xml defines
    // (weight, type, slot, armor, attack, ...), added as parsed
}

bitflags! { pub struct FloorChange: u8 {
    const DOWN; const NORTH; const SOUTH; const EAST; const WEST;
    const SOUTH_ALT; const EAST_ALT;
}}

pub fn parse_items_xml(xml: &str) -> Result<ItemsXml, ItemsXmlError>;
```

- Walks `<items><item id=.. | fromid=.. toid=..>` with nested
  `<attribute key=.. value=..>`.
- Maps the `floorChange` string values
  (`down|north|south|east|west|southalt|eastalt`) to `FloorChange` bits.
- Other attributes parsed generically into the typed bag (string/int/bool by key).
- **Tests** (skip-graceful when the gitignored reference file is absent, per the
  M2 format-test pattern): the `floorChange` string‚Üíbits table; `fromid/toid`
  range expansion; a parse of the real `items.xml` with no unknown-attr panics.

### 2. `crates/formats/src/otb.rs` ‚Äî height flag

Extract `FLAG_HAS_HEIGHT (1 << 3)` from the item flags word into a new
`ItemType.has_height: bool` (mirrors the existing `FLAG_BLOCK_SOLID` extraction).

### 3. `crates/world/src/map.rs` (`StaticMap`) ‚Äî per-tile vertical metadata

Derived once at load from the merged `ItemType` data on each tile:

```rust
floor_change: HashMap<(u16,u16,u8), FloorChange>, // union of items' flags
tile_height:  HashMap<(u16,u16,u8), u8>,          // summed item heights
```

Helpers:

- `tile_height(pos) -> u8`, `triggers_up(pos) -> bool` (`>= 3`).
- `resolve_floor_change(from: Position) -> Option<Position>` ‚Äî the
  `queryDestination` port: given a tile's `FloorChange` flags, compute the landing
  `Position` (down: `z+1` + offsets + `_ALT` + adjacent-alt lookups; up: `z-1` +
  inverse offsets).

### 4. `crates/world/src/game.rs` (`do_move`) ‚Äî vertical resolution

Port of `internalMoveCreature` ordering:

1. Compute the horizontal `destPos` (current behavior).
2. **Skip vertical resolution for diagonal steps** (`!diagonalMovement` guard).
3. Apply **Mechanic A** (height up, then down) to adjust `destPos.z`.
4. Resolve **Mechanic B**: if the stepped-onto tile has `FloorChange`, run
   `resolve_floor_change` to get the final cross-floor position.
5. Validate walkability/collision on the final tile (multi-floor aware).

`MoveResult` carries the resulting position including any `z` change so the
session/protocol layer can pick the right packet.

### 5. `crates/protocol/src/walk.rs` ‚Äî floor-change packets

- `move_up_creature(old, new, map) -> Vec<u8>` ‚Üí `0xBE` + newly revealed floor
  descriptions + the west/north corner slices.
- `move_down_creature(old, new, map) -> Vec<u8>` ‚Üí `0xBF` + revealed floors +
  east/south corner slices.
- `walk_update` gains an `old.z != new.z` branch dispatching to the above.
- **Overground‚Üîunderground boundary** (z=7‚Üî8): the old tile leaves the client's
  view, so the transition uses remove + fresh map description, not `0x6D` (per
  `sendMoveCreature`). Exact byte layout verified against `protocolgame.cpp`
  during implementation.

### 6. `crates/protocol/src/map_description.rs` ‚Äî underground mode

`get_map_description` currently hardcodes the overground stack (`for nz in
(0..=7).rev()`). Add the TFS two-mode branch (`protocolgame.cpp:633-645`): when
`center_z > 7`, iterate the `z-2 ..= z+2` band (clamped to map bounds) with the
ascending z-step; overground stays as-is.

### 7. `crates/world/src/game.rs` ‚Äî multi-floor presence

- `can_see` / `spectators` gain the underground `¬±2`-floor band (the visibility
  rule M5 deferred): same as M5 horizontally, but a viewer underground also sees
  the `¬±2` floors.
- `0xBE`/`0xBF` go **only to the moving player** (their camera changes floor).
  Spectators within the band see the mover via the M5 walk transition matrix
  (`0x6D` move / appear / remove) extended to cross-floor steps; the exact
  encoding for a cross-floor `0x6D` is verified against `sendMoveCreature`.

## Data flow ‚Äî walking down a staircase

```
session: Command::Move{id, dir}
actor:   destPos = horizontal step
         (non-diagonal) apply height up/down               // Mechanic A
         if stepped tile has FloorChange:
             destPos = resolve_floor_change(tile)           // Mechanic B
         validate walkability/collision on final tile
         if blocked: push(id, cancel_walk(dir)); done
         update position (now on a new z)
         // mover's own camera:
         if new.z != old.z:
             push(id, move_down_creature | move_up_creature) // 0xBF / 0xBE
         else:
             push(id, walk_update(...))                      // 0x6D + slices
         // spectators (union of from/to bands):
         for s in spectators: 0x6D move | appear | remove
```

## Error handling

- **Step that resolves to no valid tile** ‚Äî `cancel_walk` (`0xB5`), as today.
- **Floor-change tile with a missing destination tile** ‚Äî
  `resolve_floor_change` returns `None` ‚Üí treat as a normal (possibly blocked)
  step; never panic.
- **Underground band clamping** ‚Äî z is clamped to `[0, MAP_MAX_Z]`; no
  out-of-range tile queries.
- **Bad opcode / slow client** ‚Äî unchanged from M4/M5 (logged/ignored, kicked on
  backpressure); the actor never blocks or panics.
- `#![forbid(unsafe_code)]` and strict clippy stay green.

## Testing strategy (TDD, subagent-driven ‚Äî the M4/M5 pattern)

Pure, independently verifiable pieces:

- **items.xml**: `floorChange` string‚Üí`FloorChange` table; `fromid/toid` range
  expansion; real-file parse with no unknown-attr panic (skip-graceful).
- **otb**: `has_height` extracted for a known height-flagged item.
- **`resolve_floor_change`**: down (plain), down + directional offset, `_ALT`
  two-tile, and up (inverse) cases against hand-built tiles.
- **height up/down**: `internalMoveCreature` A-mechanic on hand-built floors.
- **`0xBE` / `0xBF`**: byte-faithful round-trip against an OTClient-faithful
  decoder (the `walk.rs` pattern).
- **underground map description**: `center_z > 7` emits the `¬±2` band; overground
  unchanged.
- **`can_see` band**: `¬±2` underground in/out; overground unchanged.

Actor / integration:

- Replay: step onto a `floorChange_down` tile ‚Üí assert `0xBF` + the new floor in
  the burst; step onto an up-stair ‚Üí assert `0xBE`.
- A spectator one floor away sees a mover change onto its floor (appear) and off
  it (remove).

**Live acceptance (gate):** real OTClient descends the Thais temple staircase
into the underground and climbs back up, rendering correctly, with a second
client seeing it cross floors ‚Äî no desync.

## Worktree setup (gotcha)

`reference/tfs/` and the data files (`items.otb`, `items.xml`, `forgotten.otbm`)
are gitignored and exist only in the primary checkout. The first implementation
step symlinks them into this worktree so tests and the live run resolve them.
`items.xml` lives alongside `items.otb` under the TFS `data/items/` tree.
